Key Findings
eBPF
EF throughput accuracy is near-perfect in eBPF — 499.98 Mbps vs a 500 Mbps target (0.004% error). HTB achieves 472.2 Mbps (5.6% below target), while No QoS delivers only 390.6 Mbps as it competes with other classes.
eBPF
RTT improvement is the most striking result — eBPF reduces EF average RTT from 3,941 µs (No QoS) to 347 µs, an 11× reduction. This suggests eBPF's classifier eliminates queueing delay for high-priority packets far more aggressively than HTB's kernel-space token bucket.
HTB
HTB delivers the best overall fairness — all three classes hit within 5% of their targets (EF 94.4%, AF 95.3%, BE 95.8%) with zero retransmits across all flows. No ECN marking or packet delay was observed. HTB is the most predictable option when all classes must be served reliably.
COST
eBPF imposes a significant CPU penalty — 49.6% average CPU utilisation vs 5.1% for HTB and 4.9% for No QoS. The iperf3 host CPU report shows 103.5% for the EF stream alone (saturating both cores). In a production environment, this overhead limits scalability with this implementation.
COST
eBPF lower-priority classes suffer congestion collapse — AF and BE receive only ~41 Mbps each (vs 300/200 Mbps targets). The 11,602 AF and 14,085 BE retransmits confirm that eBPF's strict token enforcement causes TCP to repeatedly back off, suggesting the rate-limiting parameters need tuning for multi-class balance.
OBS
eBPF ECN and delay counters confirm active congestion control — 14,196 ECN-marked packets and 57,622 delayed events across AF and BE classes, with 8,098 borrow operations. EF class shows zero in all three counters, confirming the strict priority hierarchy is functioning as designed.
